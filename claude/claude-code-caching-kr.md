# Claude Code Prompt Caching

> **근거**: `source-inspected` (Anthropic 공식 prompt caching 문서), `inferred` (KV Cache 재사용 메커니즘은 Transformer 아키텍처 표준 패턴과 LLM 서빙 인프라 일반 지식에 기반한 추정. Anthropic이 내부 구현을 공개하지 않음)
>
> **관측 시점**: 2026-03-26.

Claude Code가 Opus API를 호출할 때, 요청의 앞부분(prefix)이 이전 요청과 동일하면 캐시에서 재사용된다. 이를 Prompt Caching이라 한다.

## 무엇이 캐싱되는가

매 API 요청에서 **변하지 않는 앞부분**이 캐싱된다: (`source-inspected`)

```text
Request #1:
┌─────────────────────────────┐
│ system: CLAUDE.md + MEMORY  │ ← 캐싱됨 (매 요청 동일)
│ tools: [Read, Bash, ...]    │ ← 캐싱됨 (매 요청 동일)
│ user: "hello-slack"         │ ← 새로 처리
└─────────────────────────────┘

Request #2:
┌─────────────────────────────┐
│ system: CLAUDE.md + MEMORY  │ ← 캐시 히트! (재사용)
│ tools: [Read, Bash, ...]    │ ← 캐시 히트! (재사용)
│ user: "hello-slack"         │ ← 캐시 히트! (이전 요청과 동일)
│ assistant: [thinking, ...]  │ ← 캐시 히트! (이전 응답)
│ user: [tool_result x N]     │ ← 새로 처리
└─────────────────────────────┘

Request #3:
┌─────────────────────────────┐
│ 이전까지 전체               │ ← 캐시 히트! (앞부분 동일)
│ 새 tool_result              │ ← 새로 처리
└─────────────────────────────┘
```

API 요청의 앞부분부터 바이트 단위로 이전 요청과 비교하여, 동일한 prefix를 캐시에서 재사용한다.

## 장점: 비용 + 속도

컨텍스트 사이즈는 변하지 않는다. 1M 토큰 윈도우는 그대로이고, **같은 토큰을 다시 보내는 비용이 줄어드는 것**이다. (`source-inspected`)

| | 일반 입력 토큰 | 캐시된 입력 토큰 |
|---|---|---|
| **비용** | 100% | **10%** (90% 할인) |
| **처리 속도** | 전체 처리 | 캐시 부분 건너뜀 → **TTFT(첫 토큰 시간) 단축** |
| **컨텍스트 크기** | 변하지 않음 | 변하지 않음 |

## 누가 캐싱을 하는가

**Anthropic API 서버**(인프라)가 한다. 모델(GPU)이 아니다. (`inferred`)

```text
Claude Code                 Anthropic API Server              GPU (Model)
    │                              │                              │
    │  Request #1                  │                              │
    │  (system+tools+user)         │                              │
    │─────────────────────────────>│                              │
    │                              │  전체 토큰을 KV Cache로 변환  │
    │                              │─────────────────────────────>│
    │                              │  KV Cache 저장 (서버 메모리)  │
    │                              │<─────────────────────────────│
    │                              │                              │
    │  Request #2                  │                              │
    │  (system+tools+이전대화+새것) │                              │
    │─────────────────────────────>│                              │
    │                              │                              │
    │                              │  prefix 비교 → 캐시 히트!    │
    │                              │  저장된 KV Cache 재사용       │
    │                              │  새 토큰만 GPU로 전달         │
    │                              │─────────(새 토큰만)─────────>│
    │                              │                              │
```

모델 자체는 캐싱을 모른다. 모델 입장에서는 KV Cache가 이미 로드된 상태로 새 토큰만 처리하면 되는 것이다. 캐시 관리(저장, 비교, 만료)는 모두 API 서버 인프라의 역할이다.

## LLM 추론의 2단계

LLM 추론은 GPU에서 2단계로 진행된다: (`inferred`: Transformer 아키텍처 표준 동작)

```text
┌──────────────────────────────────────────────────────────┐
│ Phase 1: Prefill (입력 토큰 처리)                         │
│                                                          │
│ 입력: "You are Claude Code... hello-slack"  (50K 토큰)    │
│                                                          │
│ GPU가 하는 일:                                            │
│   각 토큰마다 모델의 모든 레이어를 통과시킴                  │
│   토큰 → Attention 계산 → Key, Value 벡터 생성             │
│                                                          │
│   토큰 1 → [K1, V1]                                      │
│   토큰 2 → [K2, V2]  (토큰 1도 참조)                      │
│   토큰 3 → [K3, V3]  (토큰 1,2도 참조)                    │
│   ...                                                    │
│   토큰 50000 → [K50000, V50000]  (이전 모든 토큰 참조)     │
│                                                          │
│   결과: KV Cache = [(K1,V1), (K2,V2), ..., (K50000,V50000)] │
│                                                          │
│   소요: 수초 (GPU 연산 집약적, 병렬 처리 가능)              │
└──────────────────────────────────────────────────────────┘
                         │
                         ▼
┌──────────────────────────────────────────────────────────┐
│ Phase 2: Decode (출력 토큰 생성)                          │
│                                                          │
│ GPU가 하는 일:                                            │
│   KV Cache를 참조하여 다음 토큰을 하나씩 생성               │
│                                                          │
│   KV Cache + "6개" → "채"                                │
│   KV Cache + "6개 채" → "널"                              │
│   KV Cache + "6개 채널" → "을"                            │
│   ...                                                    │
│                                                          │
│   소요: 토큰당 수십ms (순차적, 하나씩만 생성 가능)           │
└──────────────────────────────────────────────────────────┘
```

## KV Cache란

Transformer의 Attention 수식:

```text
Attention(Q, K, V) = softmax(Q × K^T / √d) × V
```

**K(Key)**와 **V(Value)**는 Attention 메커니즘에서 생성되는 벡터이다. 입력 토큰이 모델의 각 레이어를 통과할 때마다 K, V 벡터가 생성된다:

```text
입력 토큰 "hello"
    │
    ▼
┌─────────────────────┐
│ Layer 1              │
│  Q(Query)  ─┐       │
│  K(Key)    ─┼─ Attention 계산 → 출력
│  V(Value)  ─┘       │
│                     │
│  K1, V1 저장 ──────────→ KV Cache
├─────────────────────┤
│ Layer 2              │
│  K2, V2 저장 ──────────→ KV Cache
├─────────────────────┤
│ ...                  │
│ Layer N (Opus급 128+)│
│  KN, VN 저장 ──────────→ KV Cache
└─────────────────────┘
```

새 토큰을 생성할 때 **Q만 새로 계산**하고, K와 V는 이전에 계산해둔 캐시를 재사용한다. Prompt Caching은 이 KV Cache를 **API 요청 간에도 보존**하여, 같은 prefix를 다시 처리하지 않는 것이다.

```text
Request #1:  50K 토큰 → GPU가 K1..K50000, V1..V50000 계산 → 서버가 저장
Request #2:  50K + 1K 토큰 → 저장된 K,V 로드 → 새 1K만 K,V 계산
```

## 캐시 있을 때 vs 없을 때

```text
캐시 없음 (Request #1):
┌──────────────────────────────────────┐
│ Prefill: 50K 토큰 전체 처리           │  ← GPU 수초 소요
│ → KV Cache 생성                      │
│ → API 서버가 KV Cache 저장            │
├──────────────────────────────────────┤
│ Decode: 응답 토큰 생성               │
└──────────────────────────────────────┘

캐시 있음 (Request #2, prefix 50K 동일 + 새 토큰 1K):
┌──────────────────────────────────────┐
│ Prefill: 건너뜀!                     │  ← GPU 연산 없음
│ → 저장된 KV Cache를 GPU 메모리에 로드  │  ← 메모리 복사만 (빠름)
│ → 새 1K 토큰만 처리                   │  ← GPU 소량 연산
├──────────────────────────────────────┤
│ Decode: 응답 토큰 생성               │
└──────────────────────────────────────┘
```

| | 캐시 없음 | 캐시 있음 |
|---|---|---|
| GPU 연산 | 50K 토큰 × N 레이어 forward pass | 1K 토큰 × N 레이어만 |
| 병목 | **GPU 연산** (compute-bound) | **메모리 전송** (memory-bound) |
| 시간 | 수초 | 수백ms |
| 비용 | 100% | 10% (GPU 연산 거의 없고 메모리 I/O만) |

## hello-slack 실행 시 캐싱 효과

> 아래 수치는 구조 설명을 위한 추정치이다. 실제 토큰 수는 CLAUDE.md 크기, 도구 수, 검색 결과 크기에 따라 달라진다.

```text
                          전체 토큰    캐시 히트    새 토큰
Request #1:
  system (CLAUDE.md 등)     ~30K        0          30K
  tools 정의                ~20K        0          20K
  user ("hello-slack")      ~2K         0          2K
                            ~52K        0          52K    → 전액 과금

Request #2:
  이전 전체                  ~52K       52K         0     → 10% 과금
  새 tool_result xN         ~40K        0          40K    → 전액 과금

Request #3:
  이전 전체                  ~92K       92K         0     → 10% 과금
  새 tool_result (Write)     ~1K        0          1K     → 전액 과금

Request #4:
  이전 전체                  ~93K       93K         0     → 10% 과금
  새 tool_result (Bash)      ~0.5K      0          0.5K   → 전액 과금
```

캐싱 없이 vs 있을 때 대략적 비교:

```text
캐싱 없음: 52K + 92K + 93K + 93.5K = ~330K 토큰 기준 과금
캐싱 있음: 52K + (5.2K+40K) + (9.2K+1K) + (9.3K+0.5K) = ~117K 토큰 기준 과금

절약률: ~65%
```

대화가 길어질수록 캐시 히트 비율이 높아져 절약률이 더 커진다.

## 10% 비용의 이유

캐시 히트 시에도 **저장된 KV Cache를 GPU 메모리에 로드**하는 비용이 존재한다. 완전히 무료가 아닌 10%를 과금하는 이유이다. 다만 토큰을 처음부터 처리(forward pass)하는 것보다 훨씬 저렴하다. (`inferred`)

## 요약

| 질문 | 답 |
|------|---|
| 비용이 줄어드는가? | **O**. 캐시된 토큰은 10% 비용 |
| 컨텍스트 사이즈가 줄어드는가? | **X**. 1M 윈도우는 그대로 |
| 속도가 빨라지는가? | **O**. 캐시된 부분을 건너뛰어 TTFT 단축 |
| 무엇이 캐싱되는가? | 요청의 앞부분(system, tools, 이전 대화) |
| 누가 캐싱하는가? | Anthropic API 서버 (모델이 아님) |
| 왜 KV Cache인가? | Transformer Attention의 K, V 벡터를 재사용 |
| 왜 10%인가? | KV Cache를 GPU 메모리에 로드하는 비용 |
